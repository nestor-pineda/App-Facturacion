# Injection

### ¿Qué es Injection?

Injection es una vulnerabilidad que ocurre cuando una aplicación **toma datos enviados por el usuario y los interpreta como instrucciones o código**.

En lugar de tratar esa información solo como texto, el sistema la ejecuta.

👉 **Regla básica:** nunca confiar en la entrada del usuario.

---

### ¿Por qué ocurre?

Injection aparece principalmente por:

- No separar correctamente **datos** e **instrucciones**.
- Falta de validación o limpieza de los datos de entrada.
- Concatenar directamente datos del usuario en consultas o comandos.
- Confiar en que el cliente (navegador) limpia los datos.

📌 **Idea clave:** los datos del usuario siempre deben tratarse solo como datos, nunca como código.

---

### Ejemplos típicos

🔴 **Ejemplo 1: Consulta SQL vulnerable**

```sql
SELECT * FROM users WHERE id = ' + userInput;

```

Si el usuario introduce:

```
1 OR 1=1

```

👉 La consulta devuelve todos los usuarios.

🟢 **Forma correcta:**

Usar consultas parametrizadas para que el input no se ejecute como código.

---

🔴 **Ejemplo 2: XSS (Cross-Site Scripting)**

Un comentario de usuario se muestra sin validación:

```html
<script>alert('Hack');</script>

```

👉 Ese código se ejecuta en el navegador de otros usuarios.

🟢 **Forma correcta:**

Escapar el contenido antes de mostrarlo en la página.

---

### Tipos más comunes de Injection

### 🗄️ SQL Injection (SQLi)

- El atacante modifica una consulta SQL.
- Riesgos: leer, modificar o borrar datos.
- Prevención:
    - Consultas parametrizadas.
    - Validar tipos de datos.
    - Usar cuentas de base de datos con pocos privilegios.

### 🌐 Cross-Site Scripting (XSS)

- Código del atacante se ejecuta en el navegador de otros usuarios.
- Riesgos:
    - Robo de cookies y sesiones.
    - Manipulación de la interfaz.
- Tipos: reflejado, persistente y basado en DOM.
- Prevención:
    - Escapado según el contexto.
    - Validación en servidor.
    - Uso de Content Security Policy (CSP).

---

### ¿Qué puede hacer un atacante?

Si explota Injection, puede:

- Acceder o modificar datos sensibles.
- Robar sesiones e identidades.
- Ejecutar comandos del sistema.
- Insertar código malicioso persistente.
- Usar la aplicación para fraudes o ataques a terceros.

📌 **Impacto alto:** poco esfuerzo, consecuencias graves.

---

### ¿Cómo detectarlo?

### Durante el desarrollo

- Revisar consultas y plantillas donde se insertan datos del usuario.
- Confirmar uso de APIs seguras con parámetros.

### En pruebas y pentesting

- Enviar entradas inesperadas y observar el comportamiento.
- Usar análisis estático y dinámico.

### En producción

- Revisar logs con entradas extrañas.
- Detectar errores de parsing o respuestas anómalas.
- Observar contenido sospechoso en la interfaz.

---

### ¿Cómo prevenir Injection?

### Principios básicos

- Validar y normalizar datos en el servidor.
- Separar datos de instrucciones.
- Escapar el contenido según dónde se use (HTML, JS, URL).

### Buenas prácticas

- Parametrizar consultas SQL.
- Usar ORMs o query builders seguros.
- Validación estricta (listas blancas).
- Evitar `eval` o ejecución dinámica.
- Usar cuentas de base de datos con mínimo privilegio.
- Aplicar CSP y separar endpoints sensibles.

---

### ¿Qué hacer si ya está en producción?

- Limitar el acceso al endpoint vulnerable.
- Analizar logs y medir el impacto.
- Corregir el código con validación o parametrización.
- Eliminar contenido malicioso almacenado.
- Rotar credenciales y tokens si es necesario.
- Aplicar reglas WAF mientras se corrige.
- Realizar pruebas tras el parcheo.

---

### ✅ Checklist rápido para desarrolladores

Antes de desplegar, revisa:

1. ¿Valido todas las entradas en el servidor?
2. ¿Uso consultas parametrizadas?
3. ¿Aplico escapado contextual al mostrar datos?
4. ¿Evité `eval` o ejecución dinámica?
5. ¿Las cuentas de DB tienen pocos privilegios?
6. ¿Uso CSP y controles de autorización?
7. ¿Tengo pruebas contra entradas inesperadas?

---

### Conclusiones

- Injection sigue siendo muy común, pero **es fácil de prevenir**.
- La clave es **tratar los datos como datos**, nunca como instrucciones.
- Buenas prácticas de validación y parametrización evitan fallos graves.
- Probar un endpoint con entradas inesperadas es un ejercicio simple y muy útil.
